Three public corpora, three TSET configurations, five competitor formats. Every number on this page comes from running the scripts in examples/datasets/; reproduce locally with one command per section. The full discussion lives in SCALING.md.

~600 B / doc
Inline manifest overhead per document in v0.3.2. Per-doc, not per-byte.

One sentence: TSET overhead is per-document, not per-byte. The size ratio depends on your corpus's bytes-per-doc, not how many bytes you have.

Web-shaped corpora (~1 KB / doc) sit at ~1.5×. Code corpora (>5 KB / doc) sit at <1× — TSET shrinks them. Tweet-shaped corpora (~200 B / doc) sit at ~4–5× and need to be sharded before ingest. See the deal-breaker triage at the bottom.

Why does the ratio vary so much?

Profiling a TinyShakespeare shard with all v0.3.2 sections enabled (profile_size.py):

RegionBytes% of file
header (fixed)4 KB0.0%
doc store (zstd-compressed text)681 KB6.1%
view: byte-level-v1652 KB5.8%
view: whitespace-hashed-v1409 KB3.6%
section: TSMT (binary SMT)229 KB2.0%
section: TLOG (binary audit log)2.9 MB26.2%
section: TCOL (binary metadata)135 KB1.2%
manifest (canonical JSON tail)6.2 MB55.0%
footer (fixed)64 B0.0%

The JSON manifest dominates. Inside it, audit_log alone is 47.6% of the manifest and is duplicated by the binary TLOG section (the v0.3.2 transitional state — v0.4 drops the inline form). smt_present_keys is fully redundant with the document_index; that's another 7.8% of waste. Hex encoding of every 32-byte hash to 64 chars accounts for another chunk.

The "5×" headline isn't tokens or text — it's JSON manifest. v0.4's mandatory binary sections + dropped inline forms are projected to take an 11.2 MB shard down to ~7 MB without losing a single capability.

Deal-breaker triage

From real measurements, what's a real production blocker today vs. a knob:

Concern Severity Mitigation today v0.4 plan
4–5× JSONL on tiny-doc corpora deal-breaker for tweet-shaped data Shard records into larger TSET documents unchanged — fundamental to per-doc indexing
Inline duplication when binary sections enabled bloat Leave sections off until you need them Mandatory sections, inline forms dropped
Hex encoding doubles every hash in JSON minor tolerate base64 in manifest, binary in sections
Dataset Merkle root not bound to exclusion overlay gap vs. pitch Use exclusion overlay file directly; root behavior is captured by a behavior-probe test Spec extends the leaf set

Detailed discussion + the "should I use this for 5 TB?" projection is in examples/datasets/SCALING.md.
