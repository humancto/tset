Why now

Tribal-knowledge provenance is breaking.

Three forces are converging on every team that trains a model: regulators who want defensible answers, users who can demand deletion, and researchers who want to reproduce results. Today's training pipelines can't answer any of them cleanly.

01 — Regulation

The EU AI Act lands hard.

Article 10 (data governance) and Article 53 (foundation-model obligations) require detailed records of training-data sources and characteristics. Vague S3 paths and Notion docs won't satisfy an auditor.

02 — Right to erasure

GDPR Art. 17 is now operational.

Users are filing right-to-be-forgotten requests against AI companies. You need a way to prove a document was excluded from a retraining run — cryptographically, not on a spreadsheet.

03 — Reproducibility

"What did we train on?" is a real question.

Re-running a training job a year later, debugging a regression, or handing your corpus to an external auditor all require pinning the exact bytes. JSONL files in S3 mutate. Merkle roots don't.

What's in the file

Receipts, in six shapes.

Every claim TSET makes is backed by a section of the binary you can verify with one of the open implementations. Nothing is on faith.

Provable inclusion

"Yes, this document was in the training set."

A Sparse Merkle Tree commits to every doc hash. Inclusion proofs are O(log N) and verifiable against the single root in the file header. Hand the proof to your auditor; they can verify it without our code.

Provable non-inclusion

"No, this document was excluded."

Same SMT structure gives you cryptographic proofs of absence — the missing primitive most pipelines lack. Maps directly onto GDPR Art. 17 deletion requests.

Append-only audit

Every action is signed.

Ed25519-signed log of every ingest, exclusion, and tokenizer view added.

Tokenize many ways

Append a view, never re-shuffle.

One copy of the text. Add a new tokenization in place; old views stay valid.

Reproducible

Byte-identical re-tokenization.

Cross-impl conformance fixtures lock the wire format. Byte-for-byte.

Drop-in for HuggingFace datasets

Two functions. Existing pipelines untouched.

tset.hf.to_tset(hf_dataset, "out.tset") ships any HuggingFace dataset as a TSET shard with one Merkle root. tset.hf.from_tset("corpus.tset") hands it back as a regular datasets.Dataset — map, filter, train_test_split all work. 

from tset.hf import from_tset, to_tset
ds = from_tset("corpus.tset", with_tokens=True)

How it’s laid out

One self-describing binary file.

A header anchors a Merkle root over the document store and every tokenization view. Optional v0.3.2 sections (TSMT / TLOG / TCOL) live on disk, eagerly verified at Reader::open. The manifest at the tail is content-hashed back into the header.

header / footer (tail-first verifiable) TSMT — Sparse Merkle Tree (v0.3.2) TLOG — signed audit log (v0.3.2) TCOL — columnar index (v0.3.2)

Quickstart

Three commands. Real receipts.

The Rust core, Python bindings, and CLI all read and write the same binary. Pick the surface you like; the wire format is locked. 

# Convert a JSONL corpus to TSET, with on-disk binary sections (TSMT / TLOG / TCOL)
$ tset convert jsonl input.jsonl out.tset --binary-sections

# Inspect: header, manifest, signing pubkey, sections present
$ tset inspect out.tset

# Full integrity check: manifest hash, footer, Merkle root,
# audit-log signature, per-view re-tokenization byte-equality
$ tset verify out.tset
  ✓ header magic + version
  ✓ manifest_hash matches
  ✓ shard_merkle_root  9c4f2e1a…d8b371f0
  ✓ TLOG signature (Ed25519, key ed25519:a55d…)
  ✓ 2 views verified, 1,247 chunks reproduced byte-identically
from tset import Reader, Writer
from tset.tokenizers import ByteLevelTokenizer, WhitespaceTokenizer

# Write once, append tokenizer views in place
with Writer("corpus.tset") as w:
    w.add_document(b"alpha document", metadata={"lang": "en"})
    w.add_document(b"beta",           metadata={"lang": "fr"})
    w.add_subset("english", "lang = 'en'", default_weight=0.7)
    w.add_tokenizer_view(ByteLevelTokenizer())
    w.enable_binary_sections()  # opt-in v0.3.2 on-disk TSMT/TLOG/TCOL

# Stream tokens into a training step; doc_hash travels with every batch
r = Reader("corpus.tset")
for tokens, doc_hash in r.stream_tokens("byte-level-v1", batch_size=1024):
    train_step(tokens)  # numpy uint32 batch + the source doc_hash

# Generate a non-inclusion proof for a deletion request
absent = bytes.fromhex("a55d" + "00" * 30)
proof  = r.prove_non_inclusion(absent)
assert proof.verify(r.smt_root())  # True — root comes from the v0.3.2 TSMT section
use tset_core::{Reader, Writer, Dataset};
use tset_core::tokenizers::ByteLevelTokenizer;

let mut w = Writer::create("corpus.tset", None);
w.enable_binary_sections();
w.add_document(b"alpha document")?;
w.add_tokenizer_view(Box::new(ByteLevelTokenizer))?;
w.close()?;

let r    = Reader::open("corpus.tset")?;
let view = r.open_view("byte-level-v1")?;
for (tokens, doc_hash) in view.iter_per_doc()? {
    // tokens: Vec<u32>, doc_hash: [u8; 32]
    train_step(&tokens, &doc_hash);
}

let root = r.smt_root();  // 32-byte SMT root from the TSMT section

// Dataset-level non-inclusion proof (RTBF receipt)
let ds     = Dataset::open("my-corpus.tset/")?;
let absent: [u8; 32] = [0xa5; 32];
let bundle = ds.prove_non_inclusion(&absent)?;  // serde_json::Value proof bundle

No magic, no overclaiming

What this format is not.

TSET makes integrity claims, not authenticity ones. It does not prove a document came from the URL recorded in metadata, or that a model has “forgotten” any document. It produces artifacts — non-inclusion proofs, Merkle commitments, signed audit logs — that compliance workflows can use as inputs. Whether your particular use satisfies GDPR Art. 17, the EU AI Act, or any other regulation is for your counsel to determine.

See RFC §5.7 for the long version.
