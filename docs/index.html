Training data your model can prove it actually saw.

TSET is an open binary format for LLM training corpora. Store text once, tokenize many ways, bind the whole shard with a Merkle root, and honor deletion requests across every conforming reader.

Why a new format?

Training pipelines today juggle JSONL, Parquet, WebDataset, MDS, and a dozen tokenizer caches — each with its own mental model for provenance. TSET is one binary. One Merkle root. One source of truth for what every token came from.

One copy of the text

Documents are stored once, content-addressed by BLAKE3. New tokenizations are appended in place — no re-shuffling raw bytes.

Provenance you can prove

Inclusion and non-inclusion proofs against a fixed-depth Sparse Merkle Tree. Every chunk carries its own content hash.

Compliance-shaped

An exclusion overlay (binding into the dataset Merkle root) is honored by every conforming reader. The audit log is Ed25519-signed.

Reproducible by construction

Re-tokenizing the same shard on the same code yields byte-identical chunks. Cross-impl conformance fixtures lock the wire format.

How it’s laid out

A TSET shard is a self-describing binary file. The header anchors a Merkle root over the document store and every tokenization view; the manifest at the tail is content-hashed back into the header. 

┌─ MAGIC + version_minor ──────────────────────────────┐
│  shard_merkle_root  ·  manifest_hash                 │
├─ DOC STORE ──────────────────────────────────────────┤
│  zstd-compressed blocks · per-doc BLAKE3 addressed   │
├─ TOKENIZER VIEWS (one per view) ─────────────────────┤
│  bit-packed token IDs · per-chunk content_hash       │
├─ OPTIONAL on-disk sections (v0.3.2) ─────────────────┤
│  TSMT  Sparse Merkle Tree                            │
│  TLOG  Append-only audit log (Ed25519 signed)        │
│  TCOL  Columnar metadata index                       │
├─ MANIFEST (canonical JSON) ──────────────────────────┤
│  document index · views · subsets · weights          │
└─ FOOTER (mirrors header for tail-first verify) ──────┘

Write once

The Writer streams documents to disk — no in-memory doc_contents pile-up.

02

Tokenize many ways

append_tokenizer_view bolts a new view onto an existing shard without rewriting the document store.

03

Verify everywhere

Reader::open eagerly checks every section content-hash. Tampered bytes fail loudly at open.

# Inspect the shard (header, manifest, signing pubkey, sections)
$ tset inspect out.tset

# Full integrity check: manifest hash, footer, Merkle root,
# audit log signature, per-view reproducibility
$ tset verify out.tset
from tset import Reader, Writer
from tset.tokenizers import ByteLevelTokenizer, WhitespaceTokenizer

with Writer("corpus.tset") as w:
    w.add_document(b"alpha document", metadata={"lang": "en"})
    w.add_document(b"beta", metadata={"lang": "fr"})
    w.add_subset("english", "lang = 'en'", default_weight=0.7)
    w.add_tokenizer_view(ByteLevelTokenizer())
    w.enable_binary_sections()  # opt-in v0.3.2 on-disk TSMT/TLOG/TCOL

r = Reader("corpus.tset")
for tokens, doc_hash in r.stream_tokens("byte-level-v1", batch_size=1024):
    train_step(tokens)  # numpy uint32 batch + the source doc_hash

What this format is not

TSET makes integrity claims, not authenticity ones. It does not prove a document came from the URL recorded in metadata, or that a model has “forgotten” any document. It produces artifacts — non-inclusion proofs, Merkle commitments, signed audit logs — that compliance workflows can use as inputs. Whether your particular use satisfies GDPR Art. 17, the EU AI Act, or any other regulation is for your counsel to determine.

See RFC §5.7 for the long version.
